## PRD에 들어가야 할 핵심 내용

### 1. 문서 메타데이터 (헤더)
- 프로젝트명, 문서 유형, 버전, 작성일
- 목적 요약 (1~2문장)
- 참조 문서 목록

현재 PRD 상태: **완비** ✓

---

### 2. 배경 및 목적 (Why)
| 항목 | 설명 |
|------|------|
| **프로젝트 동기** | 왜 이것을 만드는가 — 해결하려는 문제 |
| **Why 체인** | 근본 원인까지 파고드는 연쇄 질문 |
| **대상 선정 근거** | 왜 이 방식/크기/범위인가 (비교 분석) |

현재 PRD 상태: **완비** ✓ (1.1~1.3)

---

### 3. Epic & Persona (누가, 무엇을 위해)
| 항목 | 설명 |
|------|------|
| **Epic 명칭** | 훈련/비즈니스 목표를 한 문장으로 |
| **훈련/비즈니스 목표** | 구체적 달성 목표 N가지 |
| **학습자/사용자 Persona** | 역할, 현재 상태, 핵심 어려움, 목표 |

현재 PRD 상태: **완비** ✓ (2.1~2.3)

---

### 4. 도메인 불변조건 (Invariants)
> PRD에서 가장 중요한 섹션 중 하나. "절대 변경할 수 없는 규칙"을 먼저 정의해야 한다.

- ID 부여 (예: INV-01)
- 조건 설명
- 위반 시 처리 방법

현재 PRD 상태: **완비** ✓ (섹션 3, INV-01~INV-09)

---

### 5. 입출력 계약 (Contract)
| 항목 | 내용 |
|------|------|
| **입력 계약** | 타입, 허용 범위, 형식 |
| **출력 계약** | 타입, 포맷, 순서, 좌표계 |
| **검증 순서** | 다단계 검증의 우선순위 규칙 |
| **완성 조건** | 성공 판정의 기준 |

현재 PRD 상태: **완비** ✓ (섹션 4)

---

### 6. 추적성 (Traceability)
> "개념이 어떻게 테스트와 컴포넌트로 연결되는가"를 보여주는 수직 체인

```
Concept → Rule(Invariant) → Use Case → Contract → Test ID → Component
```

현재 PRD 상태: **완비** ✓ (섹션 5, 14)

---

### 7. 기능 요구사항 (Functional Requirements)
| 항목 | 설명 |
|------|------|
| **FR-Bxx** | Boundary(UI) 레이어 요구사항 |
| **FR-Dxx** | Domain 로직 요구사항 |
| **FR-Rxx** | Data/Repository 요구사항 |
| 각 FR 항목마다 | 요구사항 + 제약 + **검증 가능 여부** 명시 |

현재 PRD 상태: **완비** ✓ (섹션 6)

---

### 8. 에러 정책
- 에러 코드 목록 (ID + 발생 조건 + 메시지 문구 + 처리 레이어)
- 에러 처리 원칙 (첫 번째 실패 즉시 반환, bare except 금지 등)

현재 PRD 상태: **완비** ✓ (섹션 7)

---

### 9. 아키텍처 제약
- 레이어 구조 (ECB, Clean Architecture 등)
- 의존성 방향 (허용/금지)
- 파일 구조

현재 PRD 상태: **완비** ✓ (섹션 8)

---

### 10. 인수 시나리오 (Gherkin / BDD)
> 자연어로 검증 가능한 시나리오. 개발자·비개발자 모두 이해 가능해야 함.

```gherkin
Given (전제 조건)
When (행동)
Then (기대 결과)
```

현재 PRD 상태: **완비** ✓ (섹션 9)

---

### 11. 테스트 계획
| 항목 | 설명 |
|------|------|
| 테스트 분포 | 레이어별 테스트 수 + 커버리지 목표 |
| TDD 단계별 기준 | RED/GREEN/REFACTOR 진입·완료 조건 |
| Mock 정책 | 어느 레이어를 Mock으로 대체하는가 |
| 실행 명령 | 실제 pytest 명령어 |

현재 PRD 상태: **완비** ✓ (섹션 10)

---

### 12. 비기능 요구사항
- 코드 스타일, 포매터, 타입힌트 기준
- 금지 패턴 (forbidden patterns) + 이유 + 대안
- 도메인 상수 (단일 진실 공급원)
- 테스트 패턴 (AAA, fixture scope 등)

현재 PRD 상태: **완비** ✓ (섹션 11)

---

### 13. 성공 기준 (Definition of Done)
> "언제 이 프로젝트가 완료됐다고 판단하는가"를 수치로 정의

| 기준 ID | 항목 | **목표값** | 측정 방법 |
|---------|------|-----------|-----------|
| SC-01 | 커버리지 | ≥ 95% | pytest --cov |

현재 PRD 상태: **완비** ✓ (섹션 12)

---

### 14. 범위 정의 (Scope)
- **In-Scope:** 이번 PRD에서 다루는 것
- **Out-of-Scope:** 명시적으로 제외하는 것 + 배제 근거

현재 PRD 상태: **완비** ✓ (섹션 13)

---

### 15. 부록
- User Journey 요약
- 변경 금지 항목 (Change Freeze)

현재 PRD 상태: **완비** ✓ (부록 A, B)

## 분석: 슬라이드 접근법 vs 현재 PRD

### 핵심 차이점

슬라이드의 Dual-Track과 현재 PRD의 Dual-Track은 **추상 개념은 같지만 구현 스택이 다릅니다.**

| 구분 | 슬라이드 (강의 자료) | 현재 PRD.md |
|------|-------------------|-----------------------|
| **UI Track** | React + Testing Library (JSX 기반) | Python `InputValidator` / `ResponseFormatter` |
| **Logic Track** | Python pytest | Python pytest |
| **언어** | JS(프론트) + Python(백) 혼합 | 순수 Python |
| **UX Contract 언어** | "보인다 / 안 보인다" | "errorCode 반환" |
| **Logic Rule 언어** | "허용한다 / 거부한다" | "InvalidBlankCountError 발생" |

---

### 슬라이드에서 배울 수 있는 것 — 3가지

**① UX Contract vs Logic Rule 언어 구분** (슬라이드 6.2)

현재 PRD에는 이 구분이 명시적으로 없습니다. 추가하면 명확성이 올라갑니다.

```
현재 PRD:  "빈칸 수 ≠ 2 → INVALID_BLANK_COUNT 반환"
개선 방향:
  UX Contract: "오류 메시지가 보인다"  ← UI 관점
  Logic Rule:  "탐색이 중단된다"       ← Domain 관점
```

**② Gherkin 레벨 구분** (슬라이드 7.2 — L1 Happy Path / L3 Failure Path)

현재 PRD의 Gherkin 시나리오는 그룹 분류(정상/실패/엣지)는 있지만, L0~L3 레벨 태깅이 없습니다.

**③ PRD 5조건 체크리스트** (슬라이드 7.1 하단)

> 명확성 · 측정 가능성 · 달성 가능성 · 관련성 · 완결성(예외 케이스 포함)

현재 PRD를 이 5조건으로 검토해볼 가치가 있습니다.

---

### "with MLOps"에 대한 질문

슬라이드에는 MLOps 언급이 없어서 **어떤 맥락으로 결합하고 싶은지** 좀 더 구체적으로 여쭤보고 싶습니다.

가능한 해석이 몇 가지 있습니다:

| 해석 | 의미 |
|------|------|
| A | 이 강의가 MLOps 과정의 일부이고, 같은 방법론을 적용하고 싶다 |
| B | Magic Square 프로젝트에 모델 학습/검증 파이프라인을 추가하고 싶다 |
| C | Cursor AI를 활용한 자동화(CI/CD, 테스트 자동 실행 등)를 포함하고 싶다 |

---

### 내 생각 — 솔직하게

**잘 되어 있는 것:**
- 현재 PRD는 슬라이드의 핵심 원리(Track 독립성, 계약 기반 설계)를 이미 충분히 반영하고 있습니다
- Logic Track 부분은 슬라이드보다 오히려 더 정교하게 설계되어 있습니다 (INV 체계, 추적성 등)

**보완할 수 있는 것:**
- **UX Contract / Logic Rule 언어 이중화** — 각 시나리오에 두 관점의 표현을 병기
- **Gherkin 레벨 태깅** — L0(기본)~L3(예외) 명시
- **MLOps 결합 범위** 명확화

| 항목 | 내용 |

## ② Functional Requirements

### 2.1 Dual-Track 독립성 원칙

```
UI Track (Boundary 검증)              Logic Track (Control+Entity 검증)
─────────────────────────────         ─────────────────────────────────
UX Contract 언어로 RED 작성           Logic Rule 언어로 RED 작성
Domain을 Mock으로 완전 대체           외부 의존 없음 (순수 Python)
테스트: UI-T01~T19 (19개)             테스트: BF/MF/MV/SS/SR (29개)

⚠ UI 테스트는 로직을 모르고, Logic 테스트는 UI를 모른다
```

---

### 2.2 언어 체계 — UX Contract vs Logic Rule

| UX Contract 언어 (UI Track) | Logic Rule 언어 (Logic Track) |
|---------------------------|------------------------------|
| 보인다 / 안 보인다 | 허용한다 / 거부한다 |
| 가능하다 / 불가능하다 | 유지한다 / 중단한다 |
| 활성화 / 비활성화 | 반환한다 / 차단한다 |
| 포함한다 / 포함하지 않는다 | 계산한다 / 저장한다 |

---

### 2.3 3단 매핑 표 (Scenario → UX Contract → Logic Rule)

| Scenario (Gherkin 레벨) | UX Contract | Logic Rule |
|------------------------|-------------|------------|
| 올바른 입력으로 마방진 완성 **(L1)** | `"result"` 키가 포함된 응답이 보인다 | 해가 반환된다 |
| 조합 A 실패 후 조합 B로 완성 **(L1)** | `n1 > n2`인 응답이 보인다 | 두 번째 조합이 허용된다 |
| 두 빈칸이 같은 행에 위치 **(L2)** | `r1 = r2`인 응답이 보인다 | row-major 순서가 유지된다 |
| 누락 숫자가 1과 16인 극단값 **(L2)** | 정상 응답이 보인다 | 극단값 쌍이 허용된다 |
| 격자가 4×4가 아닌 경우 **(L3)** | `"INVALID_GRID_SIZE"` 오류가 보인다 | 크기 검증이 거부된다 |
| 빈칸이 2개가 아닌 경우 **(L3)** | `"INVALID_BLANK_COUNT"` 오류가 보인다 | 탐색이 중단된다 |
| 셀 값이 0~16 범위 초과 **(L3)** | `"INVALID_CELL_VALUE"` 오류가 보인다 | 값 범위 검증이 차단된다 |
| 0 제외 중복 숫자 존재 **(L3)** | `"DUPLICATE_VALUE"` 오류가 보인다 | 중복 탐지가 거부된다 |
| 두 조합 모두 마방진 불만족 **(L3)** | `"NO_SOLUTION"` 오류가 보인다 | 풀이가 중단된다 |
| `None` 입력 **(L3)** | `"INVALID_INPUT"` 오류가 보인다 | 입력이 차단된다 |
| 격자 저장 후 로드하여 재솔빙 **(L2)** | 동일한 응답이 보인다 | 저장/로드가 유지된다 |

> ⚠ 동사로 끝나지 않는 To-Do는 테스트로 변환하지 말 것 — **판단(Decision)을 포함한 것만 변환 대상**

---

### 2.4 Gherkin 시나리오 (L0~L3 레벨 태깅)

**레벨 정의:**

| 레벨 | 의미 | 예시 |
|------|------|------|
| **L0** | 시스템 초기화 / 사전 조건 설정 | 격자 객체 생성, 저장소 초기화 |
| **L1** | Happy Path (정상 동작) | 유효 입력 → 정상 결과 반환 |
| **L2** | 경계값 / 엣지 케이스 | 극단값, 같은 행/열 빈칸, 저장-로드 왕복 |
| **L3** | 실패 / 오류 케이스 | 모든 에러 코드 7종 발생 시나리오 |

---

**L0 — 시스템 초기화**
```gherkin
# IT-L0-01 — INV-03
Scenario: 유효한 4×4 격자 객체 초기화
  Given 4행 4열의 정수 배열이 준비되어 있다
  When  MagicSquare 객체를 생성한다
  Then  객체가 정상적으로 생성된다
```

**L1 — Happy Path**
```gherkin
# IT-S01 — INV-01, INV-02, INV-07
# UX Contract: "result" 키가 포함된 응답이 보인다
# Logic Rule:  해가 반환된다
Scenario: 조합 A(작은 수 → 첫 빈칸)로 마방진 완성 (L1)
  Given 유효한 4×4 격자에 빈칸 2개가 있다
  And   작은 수를 첫 빈칸에 배치하면 마방진 조건을 만족한다
  When  solve(grid)를 호출한다
  Then  결과 포맷은 [r1, c1, n1, r2, c2, n2] 이다
  And   n1 < n2 이다
  And   완성된 격자의 모든 행·열·대각선의 합은 34이다

# IT-S02 — INV-01, INV-07
# UX Contract: n1 > n2인 응답이 보인다
# Logic Rule:  두 번째 조합이 허용된다
Scenario: 조합 B(큰 수 → 첫 빈칸)로 마방진 완성 (L1)
  Given 조합 A로는 마방진 조건을 만족하지 않는다
  And   조합 B로는 마방진 조건을 만족한다
  When  solve(grid)를 호출한다
  Then  n1 > n2 이다
  And   완성된 격자의 모든 행·열·대각선의 합은 34이다
```

**L2 — 경계값 / 엣지 케이스**
```gherkin
# IT-S03 — INV-04, INV-08
# UX Contract: r1 = r2인 응답이 보인다
# Logic Rule:  row-major 순서가 유지된다
Scenario: 두 빈칸이 같은 행에 위치 (L2)
  Given 두 빈칸이 동일한 행에 있는 유효한 격자이다
  When  solve(grid)를 호출한다
  Then  r1 = r2 이다
  And   c1 < c2 이다

# IT-S04 — INV-03
# UX Contract: 동일한 응답이 보인다
# Logic Rule:  저장/로드가 유지된다
Scenario: 격자 저장 후 로드하여 재솔빙 (L2)
  Given grid_a를 repository에 id "test-01"로 저장한다
  When  id "test-01"로 로드한 격자로 solve를 호출한다
  Then  원본 grid_a로 solve한 결과와 동일한 int[6]이 반환된다

# MF-L2-01 — INV-02, INV-09
# Logic Rule: 극단값 쌍이 허용된다
Scenario: 누락 숫자가 1과 16인 극단값 (L2)
  Given 1~16 중 1과 16이 격자에 없다
  When  MissingNumberFinder.findMissing(grid)를 호출한다

```

```gherkin
# IT-F01 — INV-03
# UX Contract: "INVALID_GRID_SIZE" 오류가 보인다
```

#### 2-1. Gherkin 레벨 태깅

```
현재 PRD 시나리오 그룹:          MLOps 요구 레벨 체계:
```

#### 2-2. UX Contract 언어 — **완전 부재**

```
현재 PRD 표현:                    MLOps UX Contract 언어:
"errorCode 반환"         ←→      "오류 메시지가 보인다"
"INVALID_BLANK_COUNT"    ←→      "빈칸 오류가 보인다"
"SolveResult 반환"       ←→      "완성! 메시지가 보인다"
```

> **현재 PRD는 100% 개발자(Logic) 언어로만 작성되어 있습니다.**  
> UX Contract 언어("보인다 / 안 보인다" 계열)가 단 한 군데도 없습니다.

#### 2-3. Logic Rule 언어 — **완전 부재**

```
현재 PRD 표현:                    MLOps Logic Rule 언어:
"InvalidBlankCountError 발생"  ←→  "탐색이 중단된다"
"NoSolutionError 발생"         ←→  "풀이가 중단된다"
"SolveResult 반환"             ←→  "해가 반환된다"
```

> 현재 PRD는 구체적 예외 클래스명으로만 표현 — 추상 동사 언어가 없습니다.

#### 2-4. 3단 매핑 표 — **완전 부재**

| 현재 PRD에 있는 것 | 현재 PRD에 없는 것 |
|------------------|------------------|

| Traceability Matrix (섹션 14) | **두 언어 체계의 교차 시점 명시** |
