## 1. 작업 배경 및 목적

이전 단계(보고서 01~04)에서 확립된 결과물을 단일 PRD로 통합하는 것이 이 단계의 목표다.

| 보고서 | 결과물 | 이 PRD로 통합된 내용 |
|--------|--------|---------------------|
| 보고서 01 | 문제 정의 | INV-01~INV-09, 입출력 계약 `int[6]`, Why 체인 |
| 보고서 02 | TDD + Clean Architecture | 71개 테스트 계획, ECB 레이어, 에러 코드 7종 |
| 보고서 03 | `.cursorrules` | 개발 원칙, 금지 패턴, Mock 정책 |
| 보고서 04 | Epic + User Journey | Epic 명칭, Persona, 5단계 Journey, 성공 기준 |
| **이 보고서** | **PRD v1.1.0** | 위 4개 통합 + MLOps Dual-Track 방법론 추가 적용 |

### PRD가 필요한 이유

> 보고서 01~04는 각각의 관점에서 설계를 정의했다.  
> PRD는 이 모든 결정을 **하나의 문서로 통합**하여 구현의 단일 진실 공급원(Single Source of Truth)이 되게 한다.

```
보고서 01 (What)
보고서 02 (How — Architecture)   ──→  PRD (통합 계약서)  ──→  구현
보고서 03 (AI 행동 규칙)
보고서 04 (Why — Business Goal)
```

## 3. MLOps 방법론 분석 및 적용 결정

### 3.1 MLOps 과정 Dual-Track 방법론 핵심 원리

MLOps 과정(6~7교시)에서 제시한 Dual-Track TDD 방법론의 핵심은 다음 두 가지다.

**원리 1 — 두 Track은 서로 다른 언어로 RED를 작성한다**

```
UI Track (Boundary 검증)              Logic Track (Control+Entity 검증)
─────────────────────────────         ─────────────────────────────────
UX Contract 언어로 RED 작성           Logic Rule 언어로 RED 작성
"보인다 / 안 보인다"                   "허용한다 / 거부한다"
"포함한다 / 포함하지 않는다"           "반환한다 / 차단한다"

⚠ UI 테스트는 로직을 모르고, Logic 테스트는 UI를 모른다
```

**원리 2 — PRD는 5-Section 구조로 작성한다**

| Section | 내용 |
|---------|------|
| ① Overview | 프로젝트 목적, 문제 정의, 성공 지표 |
| ② Functional Requirements | Gherkin L0~L3, UX Contract, Logic Rule, 3단 매핑 |
| ③ Non-Functional Requirements | 성능, 보안, 확장성 |
| ④ Technical Constraints | 언어/도구, 아키텍처, 불변조건, 계약 |
| ⑤ Out of Scope | 배제 항목 + 근거 |

### 3.2 적용 결정 — 완전 재작성 vs 보완

| 방안 | 내용 | 판단 |
|------|------|------|
| A: 완전 재작성 | PRD를 MLOps 5-Section 구조로 처음부터 재작성 | ❌ 기존 Technical Constraints 상세도 손실 우려 |
| **B: 핵심 보완** | 기존 14섹션 구조 유지 + MLOps에서 부재한 항목만 추가 | ✅ 채택 |

> **채택 근거:** 초안의 INV 체계, Traceability Matrix, 에러 코드 정책은 MLOps 강의 수준을 초과하는 정교함을 갖추고 있다. 이를 버리는 것은 오히려 PRD 품질을 낮추는 결정이다.

## 4. 갭 분석 — 초안 vs MLOps 기준

v1.0.0을 MLOps 5-Section 기준으로 분석한 결과는 다음과 같다.

### 4.1 섹션별 적합성 점수

| MLOps 섹션 | 현재 PRD 상태 | 점수 |
|-----------|-------------|------|
| ① Overview | 목적/문제/Epic 있음, 성공 지표가 섹션 12에 분리 | 70% |
| **② Functional Requirements** | **UX Contract, Logic Rule, 3단 매핑, L0~L3 태깅 완전 부재** | **35%** |
| ③ Non-Functional Requirements | 코드 품질만 있고 성능·보안·확장성 미정의 | 50% |
| ④ Technical Constraints | 내용 완전하나 섹션 분산 | 85% |
| ⑤ Out of Scope | 6개 항목, 네트워크·다중사용자 미명시 | 90% |

### 4.2 핵심 부재 항목 (우선순위별)

| 우선순위 | 부재 항목 | 영향 |
|---------|---------|------|
| 🔴 즉시 | **UX Contract / Logic Rule 이중 언어 체계** | PRD가 개발자 언어로만 작성 — 기획자·QA 소통 불가 |
| 🔴 즉시 | **Gherkin L0~L3 레벨 태깅** | 시나리오의 중요도·목적 구분 불가 |
| 🔴 즉시 | **3단 매핑 표** (Scenario → UX Contract → Logic Rule) | Track 간 연결 고리 부재 |
| 🟡 권장 | 성능 요구사항 (응답 시간) | 측정 가능성 조건 미충족 |
| 🟡 권장 | 개발 환경 명시 (Git Flow, Cursor AI) | Technical Constraints 불완전 |
| 🟢 선택 | Out-of-Scope 항목 2개 추가 | 완결성 보강 |

### 5.2 섹션 6.6 — 3단 매핑 표 설계 결정

3단 매핑 표는 **모든 시나리오를 세 가지 관점으로 동시에 표현**하는 핵심 도구다.

```
Scenario (테스트 식별)
    ↓
UX Contract (기획자·QA 관점 — "보인다/안 보인다" 언어)
    ↓
Logic Rule  (개발자·도메인 관점 — "허용한다/거부한다" 언어)
```

**설계 결정 — 매핑 대상 선정 기준:**

> ⚠ 동사로 끝나지 않는 항목은 테스트 변환 대상에서 제외한다.  
> **판단(Decision)을 포함한 문장만** UX Contract / Logic Rule로 변환한다.

이 기준에 따라 17개 시나리오를 선별하여 3단 매핑 표를 구성했다.

| 레벨 | 포함 시나리오 수 | 예시 |
|------|--------------|------|
| L0 | 2개 | 객체 초기화, 저장소 초기화 |
| L1 | 2개 | 조합 A 완성, 조합 B 완성 |
| L2 | 5개 | 같은 행/열/대각선, 극단값, 저장-로드 왕복 |
| L3 | 8개 | 에러 코드 7종 + 검증 순서 |

### 5.3 섹션 9 — Gherkin L0~L3 레벨 태깅 설계 결정

**레벨 정의 채택 근거:**

| 레벨 | 정의 | 기존 그룹과의 매핑 |
|------|------|-----------------|
| L0 | 시스템 초기화 / 사전 조건 | **신설** (기존에 없던 그룹) |
| L1 | Happy Path — 정상 동작 | Scenario Group 1 (IT-S01~S02), Group 4 일부 |
| L2 | 경계값 / 엣지 케이스 | Scenario Group 1 (IT-S03~S04), Group 5 |
| L3 | 실패 / 오류 케이스 | Scenario Group 2, 3, Group 4 일부 |

**L0 시나리오 신설 결정:**

초안에는 시스템 초기화 수준의 시나리오가 없었다. MLOps 방법론에서 L0는 "테스트 실행의 전제 조건을 검증하는 단계"로 정의된다. 이를 반영하여 다음 3개를 신설했다.

```gherkin
IT-L0-01: MagicSquare 객체 초기화  → Logic Rule: 격자 구조가 유지된다
IT-L0-02: InMemoryRepository 초기화 → Logic Rule: 저장소가 준비된다
IT-L0-03: 도메인 상수 초기값 검증   → Logic Rule: 도메인 상수가 유지된다
```
