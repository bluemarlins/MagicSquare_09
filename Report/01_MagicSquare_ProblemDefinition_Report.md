## STEP 1 — Observation (관찰)

### 현재 우리가 해결하려는 상황은 무엇인가?

우리는 **숫자들 사이의 구조적 균형 조건을 프로그래밍으로 탐색하고 검증하는 상황**에 있다.

- 1부터 16까지의 정수 16개가 존재한다.
- 이 값들을 4×4 격자에 **중복 없이 배치**해야 한다.
- 배치 결과가 아래 조건을 동시에 만족해야 한다.

| 조건 항목 | 내용 |
|-----------|------|
| 각 행(Row)의 합 | 모두 동일한 값 |
| 각 열(Column)의 합 | 모두 동일한 값 |
| 두 대각선(Diagonal)의 합 | 모두 동일한 값 |

> 이것은 단순한 숫자 배치가 아니라,
> **조합 공간 안에서 다중 제약 조건을 만족하는 해를 탐색하는 문제**이다.

### 왜 4×4 마방진 문제를 다루는가?

| 크기 | 특성 |
|------|------|
| 3×3 | 해가 본질적으로 1가지(회전·반전 제외). 탐색 난이도가 낮아 훈련 효과 미흡 |
| **4×4** | 탐색 공간이 충분히 크고, 다중 제약 조건이 동시 작용. 완전 탐색과 최적화 전략 사이의 균형점 |
| 5×5 이상 | 해의 수가 폭발적으로 증가하여 초기 학습 및 설계 검증에 부적합 |

**구조적 특성:**

- 1~16의 합 = **136**, 목표 합(Magic Sum) = **34**
- 행 4개 + 열 4개 + 대각선 2개 = **총 10개의 합산 조건이 동시에 성립**해야 한다.
- 16! ≈ **2조** 개의 순열 — 무차별 전수 탐색은 현실적이지 않다.

### 이 문제는 어떤 맥락에서 등장하는가?

| 맥락 | 연관 내용 |
|------|----------|
| **알고리즘 학습** | 백트래킹, 제약 충족 문제(CSP), 조합 최적화 |
| **프로그램 설계** | 상태 표현 방식, 조건 검증 모듈화, n×n 일반화 가능성 |
| **수학적 사고** | 합산 성질, 대칭성·회전·반전에 의한 등가 해, 구조적 패턴 인식 |

> **4×4 마방진 문제는 단순한 퍼즐이 아니다.**
> 알고리즘 설계, 상태 표현, 조건 검증, 탐색 전략이 동시에 요구되는 **복합적 설계 과제**이다.

## STEP 2 — Why #1: 왜 마방진을 완성해야 하는가?

**가정 A:** "숫자를 올바른 위치에 배치하면, 모든 합산 조건이 자동으로 충족되기 때문이다."

### 이 답이 숨기고 있는 4가지 공백

| 공백 | 드러나는 구조적 문제 |
|------|-------------------|
| **①검증 기준 미정** | "올바른 위치"는 검증 기준이 이미 정의되어 있을 때만 성립한다. "완성"의 판단 기준이 먼저 정의되어야 한다. |
| **②탐색 과정 은폐** | "자동으로 충족된다"는 말은 탐색 과정을 감춘다. 16! ≈ 2조의 공간을 어떻게 탐색하는지가 핵심이다. |
| **③해의 범위 미정** | 하나만 찾으면 되는가, 전부 열거해야 하는가. 회전·반전 등가 해는 동일한 해인가. |
| **④목적 부재** | "완성해야 한다"는 의무는 목적이 아니다. 이 프로그램이 존재하는 이유가 정의되지 않았다. |

> "완성"이라는 개념은 **검증 기준이 먼저 정의될 때** 비로소 성립한다.

## STEP 3 — Why #2: 왜 프로그램으로 구현하는가?

**가정 A:** "사람이 손으로 계산하면 실수가 생기고 시간이 오래 걸리기 때문에, 프로그램으로 자동화한다."

### 4가지 관점에서의 분석

**반복 가능성 (Repeatability)**

| 반복의 종류 | 의미 |
|------------|------|
| 동일한 조건으로 재실행 | 결과의 재현성(Reproducibility) 확보 |
| 다른 초기값으로 재실행 | 다양한 해의 탐색 |
| 조건을 바꿔 재실행 | n×n 일반화 가능성 검증 |

> 반복 가능성은 **알고리즘의 정확성이 보장될 때**만 의미가 있다. 잘못된 로직의 빠른 반복은 잘못된 결과를 증폭시킨다.

**검증 자동화 (Automated Verification)**

> 10개의 합산 조건을 즉각적이고 일관되게 검증한다. 단, 검증 자동화는 **조건의 완전한 명세**를 전제로 한다. 조건이 불완전하면 틀린 해를 옳다고 판정할 수 있다.

**오류 방지 (Error Prevention)**

| 오류의 종류 | 사람 | 프로그램 |
|------------|------|---------|
| 계산 실수 | 발생 가능 | 발생하지 않음 |
| 논리 설계 오류 | 발생 가능 | **그대로 반영됨** |
| 잘못된 가정 | 발생 가능 | **증폭되어 반복됨** |

> 프로그램은 계산 실수를 막지만, **설계 오류는 막을 수 없다**. 오류 방지는 구현이 아니라 **설계의 책임**이다.

**규칙 기반 사고 훈련 (Rule-Based Thinking)**

> 프로그램으로 구현하는 과정에서 자연어 규칙을 논리 조건으로 번역하고, 탐색의 순서와 종료 조건을 명시하며, 예외 상황을 사전에 정의해야 한다. 훈련은 **의도적으로 설계된 순서가 있을 때** 유효하다.

### 핵심 발견

> 프로그램으로 구현하는 진짜 이유는 "자동화"가 아니다.
> **"모호한 규칙을 명확한 논리로 강제 변환하는 과정"** 그 자체가 목적이다.

## STEP 4 — Why #3: 왜 TDD 방식으로 설계하려 하는가?

**가정 A:** "테스트를 먼저 작성하면, 구현이 올바른 방향으로 수렴하기 때문이다."

### 무엇이 통제되어야 하는가?

| 통제 대상 | 통제되지 않을 때의 위험 |
|-----------|----------------------|
| 숫자 중복 여부 | 1이 두 번 사용되어도 출력이 정상처럼 보인다 |
| 합산 조건의 완전성 | 행만 검증하고 열·대각선을 누락할 수 있다 |
| 경계값 처리 | 0이나 17이 격자에 들어가는 상황을 놓친다 |
| 탐색 종료 조건 | 해를 찾았는데 계속 탐색하거나, 못 찾았는데 종료된다 |

> **핵심 발견 #1:** TDD는 통제 대상을 코드 이전에 **명시적으로 강제**한다.

### 어떤 불변 조건이 존재하는가?

```
수학적 불변 조건
  ① 사용 숫자: 1 ~ 16, 각각 정확히 1회 사용
  ② Magic Sum: 모든 행, 열, 대각선의 합 = 34
  ③ 격자 크기: 정확히 4행 × 4열

논리적 불변 조건
  ④ 배치 완료 상태에서 빈 칸은 없다
  ⑤ 합산 조건은 부분이 아니라 전체가 동시에 성립해야 한다

탐색 불변 조건
  ⑥ 이미 사용한 숫자는 다시 선택되지 않는다
  ⑦ 해를 찾은 경우, 그 배치는 반드시 불변 조건 ①②③을 통과해야 한다
```

> **핵심 발견 #2:** 불변 조건은 테스트 케이스의 원천이다. TDD는 불변 조건을 코드보다 먼저 **실행 가능한 명세**로 변환하는 행위다.

### 왜 입력/출력이 명확해야 하는가?

| 구성 요소 | 불명확할 때의 문제 |
|-----------|-----------------|
| **Input** (초기 격자 상태) | 테스트 케이스를 작성할 수 없다 |
| **Action** (탐색 실행) | 무엇을 테스트하는지 알 수 없다 |
| **Output** (완성된 격자) | 성공/실패 기준을 정의할 수 없다 |

> **핵심 발견 #3:** 입력/출력을 정의하는 행위 자체가 **설계의 시작**이다. TDD는 그 시작을 강제하는 방법론이다.

## STEP 5 — 진짜 문제 정의

### 1. 표면 문제 정의 (잘못된 정의)

> "1부터 16까지의 숫자를 4×4 격자에 배치하여, 모든 행·열·대각선의 합이 34가 되도록 만드는 프로그램을 작성한다."

**이 정의가 잘못된 이유:**

| 문제점 | 설명 |
|--------|------|
| 결과 중심 | 어떤 조건이 성립해야 성공인지를 정의하지 않는다 |
| 탐색 공간 무시 | 가능한 배치가 2조 개라는 사실이 반영되지 않는다 |
| 해의 범위 미정 | 하나를 찾는 것인지, 전부를 찾는 것인지 말하지 않는다 |
| 실패 조건 부재 | 해가 없을 때 어떤 상태가 되는지 정의하지 않는다 |
| 검증 책임 불명확 | 누가, 어떤 시점에, 무엇을 검증하는지 드러나지 않는다 |

### 2. 개선된 문제 정의 (정확한 정의)

> **"주어진 불변 조건을 모두 만족하는 배치가 존재하는지를 판별하고,
> 존재한다면 그 배치를 하나 이상 완전하게 열거한다.
> 각 배치는 독립적으로 검증 가능해야 하며,
> 검증 실패는 명확한 실패 상태로 표현되어야 한다."**

| 구성 요소 | 내용 |
|-----------|------|
| **판별** | 해의 존재 여부를 결과로 표현한다 |
| **열거** | 조건을 만족하는 배치를 하나 이상 제시한다 |
| **완전성** | 각 배치는 모든 불변 조건을 동시에 충족해야 한다 |
| **독립적 검증** | 각 배치는 단독으로 검증될 수 있어야 한다 |
| **실패 명시** | 조건을 만족하지 못하는 상태는 성공으로 오인될 수 없어야 한다 |

### 3. 이 문제의 핵심 Invariant

**Invariant I — 원소 집합의 완전성**
```
격자에 배치된 숫자의 집합은 {1, 2, 3, ..., 16} 과 정확히 일치한다.
중복도 없고, 누락도 없다.
```

**Invariant II — 격자 구조의 고정성**
```
배치는 정확히 4행 × 4열 구조를 가진다.
빈 칸은 허용되지 않는다.
```

**Invariant III — 합산 조건의 동시 성립**
```
행 4개, 열 4개, 대각선 2개 — 총 10개의 합이 모두 34와 같아야 한다.
단 하나라도 34가 아닌 경우, 그 배치는 해가 아니다.
```

**Invariant IV — 조건의 동시성**
```
Invariant I, II, III은 최종 배치에서 동시에 성립해야 한다.
부분적으로 성립하는 배치는 해(解)가 아니다.
```

### 4. 우리가 실제로 훈련하려는 사고 능력

| 사고 능력 | 훈련 내용 |
|-----------|----------|
| **① 조건의 완전한 명세** | 자연어 규칙을 빠짐없이 열거하고, 각 조건의 독립성·종속성을 구분한다 |
| **② 실패 조건을 먼저 정의** | "언제 성공인가"보다 "언제 실패인가"를 먼저 묻는다. 실패를 정의하면 성공의 경계가 드러난다 |
| **③ 불변 조건과 가변 조건의 분리** | 항상 참이어야 하는 조건(Invariant)과 구현 방식에 따라 달라지는 조건(Variable)을 구분한다 |
| **④ 검증 가능한 단위로 분해** | 하나의 큰 문제를 독립적으로 참/거짓을 판별할 수 있는 단위로 나눈다 |
| **⑤ 미정의 상태를 허용하지 않음** | 모든 가능한 상태를 열거하고, 각 상태에 이름을 붙인다. 정의되지 않은 상태는 숨겨진 오류의 서식지다 |

## 종합 결론

### Why 체인 — 3번의 질문이 수렴한 지점

```
Why #1: "완성"의 기준이 먼저 정의되어야 한다
    ↓
Why #2: 설계 오류는 프로그램도 막을 수 없다
    ↓
Why #3: TDD는 그 설계 오류를 코드 이전에 잡는 방법이다
    ↓
핵심: "무엇을 만들 것인가"가 아니라
      "어떤 조건이 성립할 때 성공인가"를 먼저 정의해야 한다
```

### 이 보고서가 확립하는 것

> 이 문제의 진짜 목적은 다음과 같다:
>
> **모호한 규칙을 완전한 조건으로 번역하고,**
> **그 조건들의 동시 성립 여부를 판별 가능한 구조로 표현하며,**
> **실패를 성공으로 오인하지 않는 설계 감각을 기르는 것이다.**
>
> 마방진은 그 훈련의 재료일 뿐이다.
